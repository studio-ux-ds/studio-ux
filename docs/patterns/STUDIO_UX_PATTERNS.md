# STUDIO_UX_PATTERNS.md — Padrões de UX · UX Patterns

> Documento normativo vivo. Responde a: **como se resolve um fluxo completo (criar, buscar, excluir, autenticar, guiar por etapas…) compondo os componentes oficiais, de modo que o mesmo problema seja sempre resolvido do mesmo jeito?**
> Living normative document. Answers: **how is a full flow solved (create, search, delete, authenticate, step through…) by composing the official components, so the same problem is always solved the same way?**
> Governança: `STUDIO_UX.md`. Princípios-âncora: P2, P6, P12, P13, P14, P16, P23, P24. Peças: `components/STUDIO_UX_COMPONENT_LIBRARY.md`. Estrutura: `layouts/STUDIO_UX_LAYOUT_SYSTEM.md`.

## 0. Como ler os padrões · How to read the patterns

**PT** — Um **padrão** é uma receita de fluxo: qual sequência de telas/estados e quais componentes resolvem um problema recorrente. Enquanto o catálogo responde "que peça?", o padrão responde "como montar a peça no fluxo?". Regra de ouro (P2): **use o padrão inteiro, não pedaços** — foi projetado com todos os estados (loading/vazio/erro/sucesso — P14), o feedback (toast — P12), a permissão (P23) e as duas variações (Desktop/Mobile — P4). Recortar metade de um padrão é como inventar um componente novo: reintroduz a inconsistência que o sistema existe para matar.

**EN** — A **pattern** is a flow recipe: which sequence of screens/states and which components solve a recurring problem. While the catalog answers "which piece?", the pattern answers "how to assemble the piece in the flow?". Golden rule (P2): **use the whole pattern, not pieces** — it was designed with every state (loading/empty/error/success — P14), the feedback (toast — P12), the permission (P23) and both variations (Desktop/Mobile — P4). Cutting half a pattern is like inventing a new component: it reintroduces the inconsistency the system exists to kill.

**PT** — Cada padrão traz: quando usar · o fluxo em prosa · componentes envolvidos · estados · feedback via toast · permissão · Desktop vs Mobile · anti-padrões.

**EN** — Each pattern carries: when to use · the flow in prose · components involved · states · toast feedback · permission · Desktop vs Mobile · anti-patterns.

## 1. CRUD (criar · ler · editar) · CRUD (create · read · edit)

**PT — Quando usar:** gerenciar uma coleção de entidades (clientes, produtos, conexões). É o padrão mais comum e a espinha dorsal de um app de gestão. **Fluxo:** a lista é a casa da entidade — um `DataTable` (Desktop) / `Card` list (Mobile) com busca, filtros e paginação. "Criar" abre um formulário (`Modal` no Desktop para tarefa curta, tela dedicada para longa; `Sheet` no Mobile) com `FormField`+controles e `FormActions`. "Ler" é a tela de detalhe (`DescriptionList`, `Tabs` para seções, `Timeline` para histórico). "Editar" reusa o mesmo formulário do criar, pré-preenchido. Excluir segue o padrão Delete (§4). **Componentes:** DataTable/List, Card, FormField+inputs, FormActions, Modal/Sheet, Tabs, DescriptionList, Toast, EmptyState, Skeleton. **Estados:** lista com loading (Skeleton), vazio (EmptyState "crie o primeiro"), erro (com repetir), populada; formulário com validação inline (FormField), submitting (loading), sucesso. **Toast:** "Cliente salvo", "Falha ao salvar" (P12) — nunca banner. **Permissão (P23):** botões de criar/editar/excluir só aparecem habilitados a quem pode; o back também barra. **Desktop vs Mobile:** Desktop tabela densa + modal; Mobile cards + Sheet + FormActions fixo no rodapé. **Anti-padrões:** dois jeitos de criar a mesma entidade (P2); formulário sem estado de erro/loading (P14); ação de editar/excluir visível a quem não pode (P23); jargão nos rótulos (P11).

**EN — When to use:** manage an entity collection (customers, products, connections). The most common pattern and the backbone of a management app. **Flow:** the list is the entity's home — a `DataTable` (Desktop) / `Card` list (Mobile) with search, filters and pagination. "Create" opens a form (`Modal` on Desktop for a short task, a dedicated screen for a long one; `Sheet` on Mobile) with `FormField`+controls and `FormActions`. "Read" is the detail screen (`DescriptionList`, `Tabs` for sections, `Timeline` for history). "Edit" reuses the create form, pre-filled. Delete follows the Delete pattern (§4). **Components:** DataTable/List, Card, FormField+inputs, FormActions, Modal/Sheet, Tabs, DescriptionList, Toast, EmptyState, Skeleton. **States:** list loading (Skeleton), empty (EmptyState "create the first"), error (with retry), populated; form with inline validation (FormField), submitting (loading), success. **Toast:** "Customer saved", "Save failed" (P12) — never a banner. **Permission (P23):** create/edit/delete buttons show enabled only to those allowed; the back also blocks. **Desktop vs Mobile:** Desktop dense table + modal; Mobile cards + Sheet + FormActions pinned to the footer. **Anti-patterns:** two ways to create the same entity (P2); a form with no error/loading state (P14); edit/delete visible to those who can't (P23); jargon in labels (P11).

### 1.1 Continuidade detalhe → edição · Detail → edit continuity

**PT — Regra:** quando a edição é aberta a partir do detalhe de uma entidade, ela é uma **variação da mesma jornada**, não uma tela que parece pertencer a outro produto. A página mantém o `PageHeader`, a largura da região, o ritmo de espaço e a superfície do detalhe; troca somente leitura (`DescriptionList`, `Timeline`) pelos campos editáveis e `FormActions`. O título continua sendo a entidade (“Aluguel”), o subtítulo deixa claro o modo (“Editando despesa”) e **Voltar/Cancelar/Salvar** retornam ao detalhe, preservando inclusive o contexto de período ou filtro. Criar a entidade pode usar um fluxo mais curto; esta regra vale especificamente para o caminho detalhe → editar.

**EN — Rule:** when editing is opened from an entity detail, it is a **variation of the same journey**, not a screen that looks like it belongs to another product. The page keeps the `PageHeader`, content width, spacing rhythm, and detail surface; it only swaps read content (`DescriptionList`, `Timeline`) for editable fields and `FormActions`. The title remains the entity (“Rent”), the subtitle makes the mode clear (“Editing expense”), and **Back/Cancel/Save** return to detail, preserving even period or filter context. Creating an entity may use a shorter flow; this rule specifically applies to the detail → edit path.

## 2. Upload · Upload

**PT — Quando usar:** o usuário envia arquivo(s) (documento, imagem, planilha). **Fluxo:** o `FileUpload` declara **antes** os tipos e o tamanho aceitos; ao escolher/arrastar, cada arquivo mostra `ProgressBar` determinado; concluído, entra na lista com opção de remover; falha por arquivo é sinalizada inline + toast. **Componentes:** FileUpload, ProgressBar, List, Toast, Button. **Estados:** idle, dragover, uploading (progresso), success, error (por arquivo e por lote), disabled. **Toast:** "3 arquivos enviados", "1 arquivo excedeu o limite" (P12). **Permissão (P23):** só quem pode anexar vê a área ativa. **Desktop vs Mobile:** Desktop arrastar-e-soltar + botão; Mobile botão que abre câmera/galeria/arquivos. **Anti-padrões:** revelar limites só no erro (erro silencioso); só drag-drop sem alternativa por botão (P19); sem estado de progresso (P16); travar em 99%.

**EN — When to use:** the user uploads file(s) (document, image, spreadsheet). **Flow:** `FileUpload` states accepted types and size **up front**; on pick/drag, each file shows a determinate `ProgressBar`; when done, it joins the list with a remove option; per-file failure is signaled inline + toast. **Components:** FileUpload, ProgressBar, List, Toast, Button. **States:** idle, dragover, uploading (progress), success, error (per file and per batch), disabled. **Toast:** "3 files uploaded", "1 file exceeded the limit" (P12). **Permission (P23):** only those allowed see the active area. **Desktop vs Mobile:** Desktop drag-and-drop + button; Mobile a button opening camera/gallery/files. **Anti-patterns:** revealing limits only on error (silent error); drag-drop only with no button alternative (P19); no progress state (P16); freezing at 99%.

## 3. Download · Download

**PT — Quando usar:** o usuário exporta/baixa um recurso (relatório, planilha, PDF). **Fluxo:** um `Button`/`IconButton` dispara; se a geração é instantânea, baixa direto; se demora, mostra loading no botão e, para lotes longos, `ProgressBar` num painel; ao concluir, toast de sucesso com o que foi gerado. **Componentes:** Button, ProgressBar, Toast, Menu (quando há formatos). **Estados:** idle, generating (loading/progresso), success, error. **Toast:** "Relatório pronto para download", "Falha ao gerar o arquivo" (P12). **Permissão (P23):** exportar dado sensível respeita permissão. **Desktop vs Mobile:** Desktop baixa arquivo; Mobile pode abrir a folha de compartilhamento do sistema. **Anti-padrões:** botão "mudo" sem feedback durante a geração (P16); baixar sem dizer o que/quando ficou pronto; expor caminho/URL técnico (P11).

**EN — When to use:** the user exports/downloads a resource (report, spreadsheet, PDF). **Flow:** a `Button`/`IconButton` triggers; if generation is instant, it downloads directly; if slow, the button shows loading and, for long batches, a `ProgressBar` in a panel; on completion, a success toast naming what was generated. **Components:** Button, ProgressBar, Toast, Menu (when there are formats). **States:** idle, generating (loading/progress), success, error. **Toast:** "Report ready to download", "Failed to generate the file" (P12). **Permission (P23):** exporting sensitive data respects permission. **Desktop vs Mobile:** Desktop downloads a file; Mobile may open the system share sheet. **Anti-patterns:** a "mute" button with no feedback during generation (P16); downloading without saying what/when it's ready; exposing a technical path/URL (P11).

## 4. Delete (destrutivo · destructive) — os 5 · the 5

**PT — Quando usar:** excluir, restaurar, sobrescrever — qualquer ato irreversível. **Fluxo (P13, "os 5 patrasques"):** (1) a ação abre um **ConfirmDialog**; (2) com **aviso/disclaimer** dizendo o que acontece e que não dá para desfazer; (3) um **tooltip** explica a consequência no gatilho; (4) o confirmar tem **estado de loading** que bloqueia duplo-clique; (5) tudo atrás de um **gate de permissão** (P23) — quem não pode nem vê a ação. O botão de confirmar é `Button danger`; o foco inicial fica no cancelar. **Antes de forçar exclusão, distinga arquivar de excluir:** o que tem histórico/referência muda de estado (arquiva); um rascunho descartável pode ser excluído de verdade (herança do IA Studio — arquivar não é dogma). **Componentes:** ConfirmDialog, Button danger, Tooltip, Toast. **Estados:** open, confirming (loading), error (toast), sucesso. **Toast:** "Cliente excluído", "Não foi possível excluir" (P12). **Desktop vs Mobile:** Desktop dialog centralizado; Mobile Sheet com botões bem separados (evitar mis-tap). **Anti-padrões:** excluir sem confirmar; confirmar sem dizer a consequência; foco padrão no botão destrutivo; ConfirmDialog para tudo (fadiga); pular qualquer um dos cinco; forçar arquivar um rascunho descartável.

**EN — When to use:** delete, restore, overwrite — any irreversible act. **Flow (P13, "the 5 safeguards"):** (1) the action opens a **ConfirmDialog**; (2) with a **disclaimer** stating what happens and that it can't be undone; (3) a **tooltip** explains the consequence on the trigger; (4) confirm has a **loading state** blocking double-click; (5) all behind a **permission gate** (P23) — those who can't don't even see it. The confirm button is a `Button danger`; initial focus stays on cancel. **Before forcing deletion, distinguish archive from delete:** what has history/references changes state (archives); a disposable draft can be truly deleted (IA Studio inheritance — archiving is not dogma). **Components:** ConfirmDialog, Button danger, Tooltip, Toast. **States:** open, confirming (loading), error (toast), success. **Toast:** "Customer deleted", "Couldn't delete" (P12). **Desktop vs Mobile:** Desktop centered dialog; Mobile Sheet with well-separated buttons (avoid mis-tap). **Anti-patterns:** deleting without confirming; confirming without the consequence; default focus on the destructive button; ConfirmDialog for everything (fatigue); skipping any of the five; forcing archive on a disposable draft.

## 5. Wizard (multi-etapa · multi-step)

**PT — Quando usar:** tarefa longa que se beneficia de ser quebrada em etapas (cadastro extenso, configuração guiada, checkout). **Fluxo:** o `Stepper` mostra as etapas nomeadas e o progresso; cada etapa valida antes de avançar; voltar não perde o preenchido; a última etapa resume o que será feito antes de confirmar. **Componentes:** Stepper/Wizard, FormField+inputs, FormActions, DescriptionList (resumo), Toast, ProgressBar (opcional). **Estados:** etapa upcoming/current/completed/error; wizard in-progress/submitting/done; loading ao carregar dados de uma etapa. **Toast:** só no resultado final ("Cadastro concluído") ou em falha; erros de etapa são inline (P12/FormField). **Permissão (P23):** o acesso ao fluxo respeita permissão. **Desktop vs Mobile:** Desktop trilho de etapas visível; Mobile "Etapa 2 de 4" compacto + navegação clara e FormActions fixo. **Anti-padrões:** perder dados ao voltar; esconder em que etapa o usuário está; forçar linearidade onde pular é válido; empilhar o wizard inteiro num Modal apertado.

**EN — When to use:** a long task better broken into steps (extensive signup, guided setup, checkout). **Flow:** the `Stepper` shows named steps and progress; each step validates before advancing; going back doesn't lose input; the last step summarizes what will happen before confirming. **Components:** Stepper/Wizard, FormField+inputs, FormActions, DescriptionList (summary), Toast, ProgressBar (optional). **States:** step upcoming/current/completed/error; wizard in-progress/submitting/done; loading when fetching a step's data. **Toast:** only on the final result ("Signup complete") or failure; step errors are inline (P12/FormField). **Permission (P23):** access to the flow respects permission. **Desktop vs Mobile:** Desktop a visible step rail; Mobile compact "Step 2 of 4" + clear navigation and pinned FormActions. **Anti-patterns:** losing data on back; hiding which step the user is on; forcing linearity where skipping is valid; cramming the whole wizard into a tight Modal.

## 6. Login · Login

**PT — Quando usar:** autenticar um usuário existente. **Fluxo:** tela enxuta, foco único (P6): identidade + senha (ou método sem senha), `FormActions` com um `primary` "Entrar", link para recuperar acesso. Erros de credencial via toast (nunca revelando qual campo falhou por segurança); estados de bloqueio/2FA quando houver. **Componentes:** FormField, TextInput, Button (primary), Link, Toast, Spinner. **Estados:** default, submitting (loading), error (toast genérico), sucesso (transição para o app). **Toast:** "Não foi possível entrar. Verifique seus dados" (P12) — mensagem que não vaza qual parte falhou. **Desktop vs Mobile:** Desktop cartão centralizado no container de leitura; Mobile tela cheia, teclado certo, botão de largura total. **Anti-padrões:** dizer "senha incorreta" vs "usuário inexistente" (vaza informação); placeholder como rótulo (acessibilidade); erro em banner (P12); múltiplas ações primárias competindo.

**EN — When to use:** authenticate an existing user. **Flow:** a lean, single-focus screen (P6): identity + password (or passwordless method), `FormActions` with one `primary` "Sign in", a recover-access link. Credential errors via toast (never revealing which field failed, for security); lockout/2FA states when present. **Components:** FormField, TextInput, Button (primary), Link, Toast, Spinner. **States:** default, submitting (loading), error (generic toast), success (transition into the app). **Toast:** "Couldn't sign in. Check your details" (P12) — a message that doesn't leak which part failed. **Desktop vs Mobile:** Desktop a centered card in the reading container; Mobile full screen, right keyboard, full-width button. **Anti-patterns:** saying "wrong password" vs "no such user" (info leak); placeholder-as-label (accessibility); error in a banner (P12); multiple competing primaries.

## 7. Cadastro / Signup · Signup

**PT — Quando usar:** criar uma conta/usuário novo. **Fluxo:** se são poucos campos, uma tela; se muitos, use o Wizard (§5). Cada campo em `FormField` com validação inline e requisitos ditos **antes** do erro (ex.: regra de senha visível); `PhoneInput` para telefone (E.164). Sucesso leva ao próximo passo (verificação/onboarding). **Componentes:** FormField, TextInput, PhoneInput, Checkbox (termos), Button (primary), FormActions, Toast, Stepper (se longo). **Estados:** default, validação inline, submitting, sucesso, erro (toast). **Toast:** "Conta criada", "Este e-mail já está em uso" (P12). **Desktop vs Mobile:** Desktop coluna de leitura; Mobile tela cheia empilhada + FormActions fixo. **Anti-padrões:** requisitos de senha revelados só no erro (erro silencioso); telefone com máscara BR fixa (usar PhoneInput); termos aceitos por padrão sem ação; sem estado de erro por campo (P14).

**EN — When to use:** create a new account/user. **Flow:** few fields → one screen; many → the Wizard (§5). Each field in a `FormField` with inline validation and requirements stated **before** the error (e.g. a visible password rule); `PhoneInput` for phone (E.164). Success leads to the next step (verification/onboarding). **Components:** FormField, TextInput, PhoneInput, Checkbox (terms), Button (primary), FormActions, Toast, Stepper (if long). **States:** default, inline validation, submitting, success, error (toast). **Toast:** "Account created", "This email is already in use" (P12). **Desktop vs Mobile:** Desktop reading column; Mobile full-screen stacked + pinned FormActions. **Anti-patterns:** password rules revealed only on error (silent error); phone with a fixed BR mask (use PhoneInput); terms pre-accepted with no action; no per-field error state (P14).

## 8. Pesquisa / Search · Search

**PT — Quando usar:** encontrar itens numa coleção grande. **Fluxo:** um campo de busca (na TopBar para busca global, no topo da lista para busca local) filtra ao digitar com pequeno *debounce*; resultados atualizam com estado de loading; sem resultados → `EmptyState` de busca ("Nenhum resultado para '...'"), distinto do vazio "primeira vez". **Componentes:** TextInput/Combobox, DataTable/List, EmptyState, Skeleton, Toast (só em erro). **Estados:** idle, typing, loading, results, empty (sem resultado), error. **Permissão (P23):** a busca só retorna o que o usuário pode ver. **Desktop vs Mobile:** Desktop busca ancorada + `CommandPalette` para poder; Mobile busca em tela/Sheet com teclado. **Anti-padrões:** busca sem estado vazio de "nenhum resultado" (P14); confundir vazio de busca com vazio inicial; `IN` grande que volta vazio calado (erro silencioso — validar contra dado real); sem loading no assíncrono.

**EN — When to use:** find items in a large collection. **Flow:** a search field (in the TopBar for global search, atop the list for local search) filters on type with a small *debounce*; results update with a loading state; no results → a search `EmptyState` ("No results for '...'"), distinct from the "first time" empty. **Components:** TextInput/Combobox, DataTable/List, EmptyState, Skeleton, Toast (error only). **States:** idle, typing, loading, results, empty (no results), error. **Permission (P23):** search returns only what the user may see. **Desktop vs Mobile:** Desktop anchored search + `CommandPalette` for power; Mobile search in a screen/Sheet with the keyboard. **Anti-patterns:** search with no "no results" empty state (P14); confusing search-empty with initial-empty; a large `IN` returning empty silently (silent error — validate against real data); no loading in async.

## 9. Filtros · Filters

**PT — Quando usar:** refinar uma coleção por critérios (status, período, categoria). **Fluxo:** controles de filtro (Select/Combobox/DatePicker/SegmentedControl) acima ou ao lado da lista; filtros aplicados aparecem como `Tag` removíveis; um "limpar filtros" reseta; a lista reflete o resultado com seus estados. As **opções de filtro vêm filtradas da fonte** (só as utilizáveis — herança do IA Studio: o operador só vê o que dá para escolher). **Componentes:** Select, Combobox, DatePicker, SegmentedControl, Tag (chips), DataTable/List, EmptyState, Button (limpar). **Estados:** sem filtro, filtrando (loading), com resultado, sem resultado (EmptyState de busca), erro. **Desktop vs Mobile:** Desktop barra/painel de filtros lateral; Mobile filtros num `Sheet` com "aplicar", contagem de ativos visível. **Anti-padrões:** filtros que somem sem indicar que estão ativos; opção de filtro que não retorna nada (deveria ser filtrada da fonte); sem "limpar"; sem estado vazio quando o filtro não acha (P14).

**EN — When to use:** refine a collection by criteria (status, period, category). **Flow:** filter controls (Select/Combobox/DatePicker/SegmentedControl) above or beside the list; applied filters appear as removable `Tag`s; a "clear filters" resets; the list reflects the result with its states. **Filter options come filtered from the source** (only usable ones — IA Studio inheritance: the operator only sees what's choosable). **Components:** Select, Combobox, DatePicker, SegmentedControl, Tag (chips), DataTable/List, EmptyState, Button (clear). **States:** unfiltered, filtering (loading), with results, no results (search EmptyState), error. **Desktop vs Mobile:** Desktop a filter bar/side panel; Mobile filters in a `Sheet` with "apply", a visible active count. **Anti-patterns:** filters that vanish without showing they're active; a filter option that returns nothing (should be filtered from the source); no "clear"; no empty state when the filter finds nothing (P14).

## 10. Paginação · Pagination

**PT — Quando usar:** percorrer uma coleção grande de dados operacionais. **Fluxo:** a `Pagination` mostra a página atual e o **total real** (nunca escondido — anti-erro-silencioso), com tamanho de página escolhível; trocar de página mostra loading no conteúdo, o shell fica estável. Para dados operacionais preferimos paginação a scroll infinito (previsibilidade, contagem, endereçabilidade). **Componentes:** Pagination, DataTable/List, Skeleton. **Estados:** default, página ativa, loading (troca), disabled (primeira/última). **Desktop vs Mobile:** Desktop controle numérico; Mobile "carregar mais" ou paginação compacta. **Anti-padrões:** paginação sem total; misturar scroll infinito com paginação; `rp`/limite que trunca sem avisar (erro silencioso); página ativa só por cor (P17).

**EN — When to use:** traverse a large collection of operational data. **Flow:** `Pagination` shows the current page and the **real total** (never hidden — anti-silent-error), with a choosable page size; changing page shows loading in the content, the shell stays stable. For operational data we prefer pagination to infinite scroll (predictability, count, addressability). **Components:** Pagination, DataTable/List, Skeleton. **States:** default, active page, loading (change), disabled (first/last). **Desktop vs Mobile:** Desktop numeric control; Mobile "load more" or compact pagination. **Anti-patterns:** pagination with no total; mixing infinite scroll with pagination; a limit/`rp` truncating silently (silent error); active page by color only (P17).

## 11. Loading · Loading

**PT — Quando usar:** sempre que há espera (P16 — nenhum clique fica "mudo"). **Fluxo:** escolha o indicador pela natureza da espera: `Skeleton` quando a forma do resultado é conhecida (telas/listas); `Spinner` para ação pontual curta (botão, área pequena); `ProgressBar` para tarefa com fim conhecido (upload/exportação). O shell permanece estável durante o loading (a navegação não pisca). **Componentes:** Skeleton, Spinner, ProgressBar, Button (loading). **Estados:** loading → success | error (sempre há saída; loading nunca é eterno). **Desktop vs Mobile:** ambos; Mobile evita spinners longos em tela cheia — prefira Skeleton. **Anti-padrões:** loading eterno sem virar erro (erro silencioso); Spinner onde caberia Skeleton (salto visual); ação sem nenhum feedback (P16); animação de loading chamando mais atenção que o dado (P1).

**EN — When to use:** whenever there's a wait (P16 — no click stays "mute"). **Flow:** choose the indicator by the wait's nature: `Skeleton` when the result's shape is known (screens/lists); `Spinner` for a short one-off action (button, small area); `ProgressBar` for a known-end task (upload/export). The shell stays stable during loading (navigation doesn't flicker). **Components:** Skeleton, Spinner, ProgressBar, Button (loading). **States:** loading → success | error (there's always an exit; loading is never eternal). **Desktop vs Mobile:** both; Mobile avoids long full-screen spinners — prefer Skeleton. **Anti-patterns:** eternal loading that never becomes an error (silent error); a Spinner where Skeleton fits (visual jump); an action with no feedback (P16); a loading animation drawing more attention than the data (P1).

## 12. Skeleton · Skeleton

**PT — Quando usar:** carregar uma tela/lista/card cuja **estrutura é conhecida**. **Fluxo:** enquanto o dado chega, mostra-se o placeholder na forma exata do conteúdo (não um bloco genérico), reduzindo o salto de layout; ao chegar o dado, troca-se sem "pulo". **Componentes:** Skeleton (espelhando DataTable/Card/DescriptionList). **Estados:** único, animado, discreto; respeita `prefers-reduced-motion`. **Desktop vs Mobile:** espelha a composição de cada produto (linhas no Desktop, cards no Mobile). **Anti-padrões:** skeleton que não parece com o resultado; pulsação forte (P1); skeleton que nunca termina (deve virar erro após timeout); usar para ação pontual (é Spinner).

**EN — When to use:** loading a screen/list/card whose **structure is known**. **Flow:** while data arrives, show a placeholder in the content's exact shape (not a generic block), reducing the layout jump; when data lands, swap without a "jump". **Components:** Skeleton (mirroring DataTable/Card/DescriptionList). **States:** single, animated, discreet; respects `prefers-reduced-motion`. **Desktop vs Mobile:** mirrors each product's composition (rows on Desktop, cards on Mobile). **Anti-patterns:** a skeleton unlike the result; a strong pulse (P1); a skeleton that never ends (should become an error after timeout); using it for a one-off action (that is Spinner).

## 13. Empty State · Empty State

**PT — Quando usar:** não há nada para mostrar (P14). **Fluxo:** distinga os casos: **primeira vez** ("Você ainda não tem clientes" + ação "Criar o primeiro") vs. **busca/filtro sem resultado** ("Nenhum resultado para esses filtros" + "limpar"). O `EmptyState` explica por que está vazio e oferece o próximo passo, em linguagem humana (P11). **Componentes:** EmptyState, Button (ação), Link. **Estados:** variações primeira-vez e sem-resultado. **Permissão (P23):** a ação sugerida só aparece a quem pode fazê-la. **Desktop vs Mobile:** centralizado no container de leitura; ação com alvo adequado. **Anti-padrões:** tela em branco sem explicação; confundir vazio com erro; oferecer "criar" a quem não tem permissão; texto técnico.

**EN — When to use:** there's nothing to show (P14). **Flow:** distinguish the cases: **first time** ("You don't have customers yet" + "Create the first") vs. **search/filter with no results** ("No results for these filters" + "clear"). `EmptyState` explains why it's empty and offers the next step, in human language (P11). **Components:** EmptyState, Button (action), Link. **States:** first-time and no-results variants. **Permission (P23):** the suggested action shows only to those who can do it. **Desktop vs Mobile:** centered in the reading container; action with a proper target. **Anti-patterns:** a blank screen with no explanation; confusing empty with error; offering "create" to someone without permission; technical text.

## 14. Erro · Error

**PT — Quando usar:** algo falhou (carregar, salvar, conectar). **Fluxo:** o feedback depende do escopo. **Ação pontual** (salvar, enviar) → `Toast` de erro na língua do usuário (P12) com o que fazer. **Falha ao carregar uma região** → estado de erro **na área de conteúdo** com uma ação de "tentar de novo"; o shell nunca some (o usuário não fica preso). **Validação de campo** → mensagem no `FormField` (não toast). Nunca engolir a falha e mostrar um fallback plausível sem sinal (a regra anti-erro-silencioso do IA Studio) — se falhou, o usuário sabe. **Componentes:** Toast, Button (repetir), estado de erro na área de conteúdo, FormField. **Estados:** erro pontual (toast), erro de região (com repetir), erro de campo (inline). **Desktop vs Mobile:** ambos; Mobile garante que a ação de repetir é alcançável. **Anti-padrões:** erro em banner ou alert nativo (P12); fallback silencioso que finge sucesso; mensagem técnica críptica (P11); esconder o erro atrás de um número "plausível".

**EN — When to use:** something failed (load, save, connect). **Flow:** feedback depends on scope. **One-off action** (save, send) → an error `Toast` in the user's language (P12) with what to do. **Region load failure** → an error state **in the content area** with a "try again" action; the shell never disappears (the user isn't trapped). **Field validation** → a message in the `FormField` (not a toast). Never swallow the failure and show a plausible fallback with no signal (IA Studio's anti-silent-error rule) — if it failed, the user knows. **Components:** Toast, Button (retry), a content-area error state, FormField. **States:** one-off error (toast), region error (with retry), field error (inline). **Desktop vs Mobile:** both; Mobile ensures the retry action is reachable. **Anti-patterns:** error in a banner or native alert (P12); a silent fallback faking success; a cryptic technical message (P11); hiding the error behind a "plausible" number.

## 15. Offline · Offline

**PT — Quando usar:** a conexão caiu ou a operação não pôde ser concluída por rede (mais relevante no Mobile). **Fluxo:** sinaliza o estado offline de forma persistente (aqui um `Banner` de uso restrito é legítimo — condição contínua, não feedback de ação); ações que exigem rede ficam desabilitadas com motivo (tooltip); ao voltar a conexão, sinaliza a recuperação e sincroniza; dados em cache são mostrados marcados como "pode estar desatualizado" (nunca dado velho fingindo ser novo). **Componentes:** Banner (restrito), Toast (ao reconectar), Button (desabilitado com motivo), Skeleton. **Estados:** online, offline, reconectando, sincronizando, erro de sync. **Desktop vs Mobile:** mais crítico no Mobile; Desktop também trata perda de rede. **Anti-padrões:** falhar calado sem dizer que está offline; mostrar cache velho como atual (erro silencioso); ação de rede clicável que só falha depois (P23/P16); usar Toast para o estado contínuo (é Banner).

**EN — When to use:** the connection dropped or an operation couldn't complete due to network (most relevant on Mobile). **Flow:** signal the offline state persistently (here a restricted-use `Banner` is legitimate — an ongoing condition, not action feedback); network-requiring actions are disabled with a reason (tooltip); when connection returns, signal recovery and sync; cached data is shown marked "may be outdated" (never stale data pretending to be fresh). **Components:** Banner (restricted), Toast (on reconnect), Button (disabled with reason), Skeleton. **States:** online, offline, reconnecting, syncing, sync error. **Desktop vs Mobile:** most critical on Mobile; Desktop also handles network loss. **Anti-patterns:** failing silently without saying it's offline; showing stale cache as current (silent error); a clickable network action that only fails later (P23/P16); using a Toast for the ongoing state (that is a Banner).

## 16. Permissões · Permissions

**PT — Quando usar:** sempre que uma ação/vista depende do que o usuário pode fazer (P23). **Fluxo:** a interface é **moldada pela permissão** — o que o usuário não pode fazer, ele não vê habilitado (padrão `can()` no front + gate no back). Uma ação sem permissão nunca aparece "clicável e depois nega". Quando faz sentido mostrar que existe mas está bloqueado, o controle fica desabilitado **com motivo** (tooltip). Degradação graciosa: se a config/infra está quebrada, encaminha/avisa em vez de erro críptico (herança do IA Studio). **Componentes:** Button/IconButton (ocultos ou desabilitados com motivo), Menu (itens filtrados), EmptyState (quando a seção inteira é vedada), Tooltip. **Estados:** permitido (habilitado), vedado (oculto), vedado-mas-visível (desabilitado com motivo). **Desktop vs Mobile:** idêntico em intenção; ambos escondem/desabilitam conforme permissão. **Anti-padrões:** botão clicável que só nega no clique (P23); esconder sem explicar onde a presença seria esperada; expor ação destrutiva a quem não pode (viola P13 também); confiar só no front (o back sempre barra).

**EN — When to use:** whenever an action/view depends on what the user can do (P23). **Flow:** the interface is **shaped by permission** — what the user cannot do, they don't see enabled (front `can()` + back gate). An unpermitted action never appears "clickable then denied". When it makes sense to show it exists but is blocked, the control is disabled **with a reason** (tooltip). Graceful degradation: if config/infra is broken, route/warn instead of a cryptic error (IA Studio inheritance). **Components:** Button/IconButton (hidden or disabled with reason), Menu (filtered items), EmptyState (when the whole section is barred), Tooltip. **States:** allowed (enabled), barred (hidden), barred-but-visible (disabled with reason). **Desktop vs Mobile:** identical in intent; both hide/disable per permission. **Anti-patterns:** a clickable button that only denies on click (P23); hiding without explanation where presence is expected; exposing a destructive action to those who can't (also violates P13); trusting the front only (the back always blocks).

## 17. Auditoria (histórico / trilha) · Audit (history / trail)

**PT — Quando usar:** o consumidor precisa mostrar quem fez o quê, quando e o que mudou (P24). **Fluxo:** a `Timeline` expõe os eventos em ordem consistente, cada um com autor (`Avatar`), carimbo de tempo legível (data absoluta acessível, não só "há 2h") e a mudança em **linguagem humana** ("mudou o status de Ativo para Suspenso"), nunca um diff cru/JSON. O Studio UX dá a forma; o sistema consumidor dá o dado (P24). Para muitos eventos, agrupa por dia e pagina. **Componentes:** Timeline, Avatar, Badge (tipo de evento), EmptyState ("sem histórico"), Skeleton, Pagination. **Estados:** loading (Skeleton), empty (sem histórico), loaded, muitos eventos (paginado/agrupado), erro. **Permissão (P23):** ver auditoria respeita permissão; dados sensíveis no diff seguem a regra de redação do consumidor. **Desktop vs Mobile:** Desktop trilho lateral com marcos; Mobile empilhado compacto. **Anti-padrões:** mostrar JSON/diff técnico em vez de frase humana (P11); ordem inconsistente; tempo só relativo sem absoluto acessível; auditoria sem estado vazio (P14).

**EN — When to use:** the consumer needs to show who did what, when and what changed (P24). **Flow:** the `Timeline` exposes events in a consistent order, each with an author (`Avatar`), a readable timestamp (accessible absolute date, not only "2h ago") and the change in **human language** ("status changed from Active to Suspended"), never a raw diff/JSON. Studio UX gives the form; the consuming system gives the data (P24). For many events, group by day and paginate. **Components:** Timeline, Avatar, Badge (event type), EmptyState ("no history"), Skeleton, Pagination. **States:** loading (Skeleton), empty (no history), loaded, many-events (paginated/grouped), error. **Permission (P23):** viewing audit respects permission; sensitive diff data follows the consumer's redaction rule. **Desktop vs Mobile:** Desktop a side rail with markers; Mobile compact stacked. **Anti-patterns:** showing JSON/technical diff instead of a human sentence (P11); inconsistent order; relative-only time with no accessible absolute; audit with no empty state (P14).
